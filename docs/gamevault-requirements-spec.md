# GameVault 游戏愿望单与游玩进度分析系统需求文档

## 1. 概述

### 1.1 项目背景

随着数字游戏发行平台的发展，玩家购买和收藏游戏的渠道逐渐增多，例如 Steam、Epic、PlayStation、Xbox、Switch 等平台。很多玩家会同时维护“想玩游戏”“已购买游戏”“正在游玩游戏”“已通关游戏”等不同状态的游戏记录。如果仅依靠平台自带收藏夹或普通表格记录，往往难以统一管理游戏状态、游玩时长、购买价格、评分评价、平台分类和游玩日志。

GameVault 游戏愿望单与游玩进度分析系统旨在为玩家提供一个集中化的游戏管理平台。用户可以在系统中维护个人游戏库和愿望单，记录游戏状态、购买价格、期望价格、评分、评价和游玩时长，并通过统计图表分析自己的游戏收藏结构、消费情况和游玩习惯。

本系统以数据库课程综合实验为背景，重点体现 Web 应用系统的需求分析、数据库设计、前后端开发、数据增删改查、统计分析和现场演示能力。系统采用 Python 后端、MySQL 数据库和 React 前端，并通过 Docker Compose 实现一键启动。

### 1.2 系统说明

本系统主要面向普通游戏玩家。用户可以注册登录系统，进入个人游戏管理页面，对游戏信息、个人游戏库、愿望单、平台、类型、标签和游玩日志进行管理。系统提供首页仪表盘、游戏库、愿望单、游玩日志、分类管理、统计分析和个人资料等功能模块。

系统主要功能包括：

1. 用户注册、登录和个人资料维护。
2. 游戏基础信息的添加、查询、修改和删除。
3. 个人游戏库和愿望单管理。
4. 游戏状态、优先级、价格、评分、时长和评价维护。
5. 平台、类型、标签分类管理。
6. 游玩日志添加、编辑、查询和删除。
7. 游戏详情查看。
8. 平台分布、类型分布、状态分布、月度趋势和榜单统计。
9. 数据库可视化查看和 Docker 一键运行。

## 2. 需求分析

### 2.1 系统需求

GameVault 系统需要满足以下需求：

第一，系统应支持用户通过账号登录，保证不同用户只能管理自己的个人游戏库和游玩日志。

第二，系统应支持游戏信息管理，包括游戏名称、原始名称、封面地址、开发商、发行商、发行日期和简介等基础数据。

第三，系统应支持个人游戏库管理。用户可以将游戏加入自己的游戏库，设置游戏状态、愿望单优先级、购买价格、期望价格、评分、游玩时长、开始日期、通关日期和个人评价。

第四，系统应支持平台、类型和标签管理。游戏可以同时关联多个平台、多个类型和多个标签，从而体现数据库多对多关系。

第五，系统应支持游玩日志管理。用户可以为个人游戏库中的游戏添加单次游玩记录，记录游玩日期、游玩时长、进度说明和备注。

第六，系统应支持统计分析。用户可以查看游戏总数、愿望单数量、正在玩数量、已通关数量、总消费、总游玩时长、平台分布、类型分布、状态分布、月度新增趋势、月度游玩时长趋势、评分榜和游玩时长榜。

第七，系统应具有良好的页面展示效果，能够支持课程现场演示，并能通过 Docker Compose 快速启动完整环境。

### 2.2 数据需求

基于系统需求分析，本系统需要实现以下数据管理需求：

1. 用户信息的输入、查询和修改，包括用户名、邮箱、加密密码、昵称、头像、简介和角色。
2. 游戏信息的输入、查询、修改和删除，包括游戏名称、原始名称、封面、开发商、发行商、发行日期和简介。
3. 平台信息的输入、查询、修改和删除，包括平台名称和图标标识。
4. 类型信息的输入、查询、修改和删除，包括类型名称和类型说明。
5. 标签信息的输入、查询、修改和删除，包括标签名称和标签颜色。
6. 用户游戏库信息的输入、查询、修改和删除，包括用户编号、游戏编号、游戏状态、优先级、购买价格、期望价格、评分、游玩时长和评价。
7. 游玩日志信息的输入、查询、修改和删除，包括用户游戏库编号、游玩日期、游玩时长、进度说明和备注。
8. 游戏与平台、游戏与类型、游戏与标签之间多对多关系数据的维护。
9. 统计分析数据的查询，包括数量统计、分布统计、趋势统计和排行榜数据。

## 4. 数据库设计

### 4.1 概念结构设计

系统主要实体包括用户、游戏、平台、类型、标签、用户游戏库、游玩日志。

实体关系如下：

1. 一个用户可以拥有多条用户游戏库记录。
2. 一个游戏可以被多个用户加入个人游戏库。
3. 一个用户游戏库记录可以拥有多条游玩日志。
4. 一个游戏可以关联多个平台，一个平台也可以关联多个游戏。
5. 一个游戏可以关联多个类型，一个类型也可以关联多个游戏。
6. 一个游戏可以关联多个标签，一个标签也可以关联多个游戏。

系统整体 E-R 图见 `docs/er-diagram.md`。

### 4.2 逻辑结构设计

#### 4.2.1 关系模型

根据概念结构设计，转换为如下关系模型：

用户(users)：用户编号、用户名、邮箱、密码哈希、昵称、头像地址、个人简介、角色、创建时间、更新时间。

游戏(games)：游戏编号、游戏名称、原始名称、封面地址、开发商、发行商、发行日期、游戏简介、创建用户、创建时间、更新时间。

平台(platforms)：平台编号、平台名称、平台图标、创建时间。

类型(categories)：类型编号、类型名称、类型说明、创建时间。

标签(tags)：标签编号、标签名称、标签颜色、创建时间。

用户游戏库(user_games)：记录编号、用户编号、游戏编号、状态、优先级、购买价格、期望价格、评分、游玩时长、开始日期、通关日期、评价、创建时间、更新时间。

游玩日志(play_sessions)：日志编号、用户游戏库记录编号、游玩日期、本次游玩时长、进度说明、备注、创建时间。

游戏平台关联(game_platforms)：游戏编号、平台编号。

游戏类型关联(game_categories)：游戏编号、类型编号。

游戏标签关联(game_tags)：游戏编号、标签编号。

#### 4.2.2 细化表结构

为便于程序实现，系统采用英文表名和字段名。核心表结构如下：

| 表名 | 主键 | 主要外键 | 说明 |
| --- | --- | --- | --- |
| users | id | 无 | 用户账号和个人资料 |
| games | id | created_by | 游戏基础信息 |
| platforms | id | 无 | 游戏平台 |
| categories | id | 无 | 游戏类型 |
| tags | id | 无 | 游戏标签 |
| user_games | id | user_id、game_id | 用户个人游戏库 |
| play_sessions | id | user_game_id | 单次游玩日志 |
| game_platforms | game_id、platform_id | game_id、platform_id | 游戏和平台关联 |
| game_categories | game_id、category_id | game_id、category_id | 游戏和类型关联 |
| game_tags | game_id、tag_id | game_id、tag_id | 游戏和标签关联 |

### 4.3 数据库完整性设计

1. 实体完整性：各主表均设置主键 `id`，关联表使用联合主键。
2. 参照完整性：外键用于约束用户、游戏、平台、类型、标签和日志之间的引用关系。
3. 用户名和邮箱唯一，平台名称、类型名称、标签名称唯一。
4. `user_games` 中 `user_id` 与 `game_id` 设置联合唯一约束，避免同一用户重复收藏同一游戏。
5. 删除平台、类型、标签时，如果已被游戏关联，则系统拒绝删除并给出提示。

## 5. 功能需求规定

### 5.1 用户认证模块

用户可以注册账号、登录系统、查看当前用户信息和修改个人资料。系统通过 Token 识别当前用户，未登录用户不能访问个人游戏库、游玩日志和统计接口。

### 5.2 游戏管理模块

用户可以新增、查询、修改和删除游戏基础信息。新增和编辑游戏时，可以选择平台、类型和标签。

### 5.3 游戏库与愿望单模块

用户可以将游戏加入个人游戏库，设置状态、优先级、购买价格、期望价格、评分、游玩时长和评价。系统支持表格视图和卡片视图，并支持按名称搜索和按状态筛选。

### 5.4 游玩日志模块

用户可以添加、编辑、查询和删除游玩日志。添加或修改日志时，系统会同步更新对应游戏的总游玩时长。

### 5.5 分类管理模块

用户可以维护平台、类型和标签。为保证数据完整性，系统不允许直接删除已关联游戏的分类数据。

### 5.6 统计分析模块

系统提供以下统计分析功能：

1. 游戏总数、愿望单数量、正在玩数量、已通关数量。
2. 总消费金额和总游玩时长。
3. 平台分布、类型分布和状态分布。
4. 月度新增游戏趋势。
5. 月度游玩时长趋势。
6. 平台消费占比。
7. 评分榜和游玩时长榜。

## 7. 性能与安全需求

### 7.1 性能需求

1. 常规列表查询应在较短时间内返回结果。
2. 搜索、筛选、统计接口应能满足课程演示数据规模。
3. 系统前后端通过 API 分离，页面切换和数据刷新应保持流畅。

### 7.2 安全需求

1. 用户密码不以明文保存，系统保存加密后的密码哈希。
2. 需要登录后才能访问个人游戏库、游玩日志和统计接口。
3. 用户只能访问和修改自己的个人游戏数据。
4. `.env` 等敏感配置文件不提交到代码仓库。

## 8. 测试需求

系统需要进行功能测试、接口测试、数据库测试和页面测试。

测试重点包括：

1. 用户注册、登录和鉴权。
2. 游戏新增、编辑、删除和详情查看。
3. 游戏库状态修改和愿望单筛选。
4. 平台、类型、标签新增和删除保护。
5. 游玩日志添加、编辑和删除。
6. 统计接口返回数据是否正确。
7. Docker 一键启动是否成功。

后端使用 `pytest` 和 `pytest-cov` 进行接口测试，当前测试覆盖率达到 90% 以上。

## 9. 总结

GameVault 游戏愿望单与游玩进度分析系统围绕游戏玩家的个人游戏管理需求，完成了用户认证、游戏管理、愿望单管理、分类管理、游玩日志、统计分析和数据库可视化展示等功能。系统数据库设计包含一对多和多对多关系，能够体现数据库课程中实体完整性、参照完整性、数据字典和关系模型设计等核心内容。
